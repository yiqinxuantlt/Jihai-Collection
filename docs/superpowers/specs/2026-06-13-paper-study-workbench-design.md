# 纸面书房三栏工作台与编辑器精修设计

日期：2026-06-13

## 1. 背景

当前应用已经具备桌面优先的三栏工作台和独立编辑页，但视觉层次仍偏功能化：左右栏的面板感略重，中栏与编辑器的主次关系还不够明确，中文排版的书写感也没有完全建立起来。

本次设计以现有产品结构为基础，不改产品定位、不更换技术栈，目标是在保留稳定三栏工作流的前提下，把界面收敛到更安静、更典雅、更适合长时间阅读与写作的“纸面书房”气质。

## 2. 已确认方向

- 整体气质：纸面书房
- 编辑器密度：偏宽松写作
- 三栏关系：左右栏轻微框定，但明确退后于中间主舞台
- 推荐方案：书房工作台

## 3. 目标

- 让桌面三栏工作台更安静、稳定、耐看，符合知识整理而非内容消费的定位
- 让编辑器成为最强视觉中心，明显提升写作沉浸感和中文阅读舒适度
- 统一标题、正文、引文、元信息、标签、选中态的视觉语言
- 优化细节显示效果，包括卡片层次、导航状态、边注信息、工具条、保存状态和空状态
- 在不增加大型依赖的前提下，基于现有 React 组件与样式完成精修

## 4. 非目标

- 不改为营销首页或社交信息流布局
- 不引入大面积高饱和渐变、玻璃拟态、霓虹色或夸张动效
- 不替换现有 Tiptap 编辑器方案
- 不改动后端接口、数据结构、导入导出协议或核心业务流程

## 5. 整体视觉概念

界面应像一张展开的书房工作台：

- 页面外层像温暖桌面，层次轻、对比低、边界稳
- 左栏像索引抽屉，负责目录与入口
- 中栏像工作台中心区域，承担浏览、阅读、选择与写作前的过渡
- 右栏像纸页边注，提供当前书籍、记录、标签和辅助操作
- 进入编辑页后，中间纸页成为唯一主角，其余元素只做安静陪衬

视觉重心必须从“控件和容器”转向“文字和内容”。

## 6. 布局设计

## 6.1 工作台页

- 保留三栏稳定结构，不做大幅跳动式切换
- 外层工作台宽度保持桌面优先，整体控制在 `1200px` 到 `1440px` 的舒适区间
- 左栏宽度建议收敛到 `248px` 到 `268px`
- 右栏宽度建议控制在 `312px` 到 `336px`
- 中栏保持弹性主舞台，优先承载书籍卡片、笔记列表、标签索引与时间线
- 三栏边界主要通过低对比底色、细边框和轻阴影建立，不依赖厚重卡片堆叠

## 6.2 编辑页

- 保留双区布局：中间纸页 + 右侧边注整理区
- 编辑页中间纸页最大宽度建议控制在 `760px` 到 `820px` 的正文舒适带
- 纸页上下留白加大，保证标题、引文、正文之间有明确呼吸感
- 右侧边注区保留 sticky 行为，但弱化“后台表单面板”观感
- 在窄屏下保留可用性，优先保障纸页阅读与输入，不强求桌面完整三栏体验

## 7. 字体与排版

## 7.1 字体分工

- UI 字体：继续使用 `PingFang SC`、`Microsoft YaHei UI`、`Segoe UI` 一类清晰耐读的系统无衬线
- 正文与书名：使用 `Noto Serif SC`、`Source Han Serif SC`、`Songti SC` 一类稳定的中文衬线
- 少量书写感点缀：仅在题眼、引文提示、小型状态文本中谨慎使用 `LXGW WenKai`、`Kaiti SC` 一类字体；不允许大面积铺满

## 7.2 页面文字层级

- 页面主标题：`30px` 到 `34px`，使用衬线字体，拉开呼吸感
- 卡片标题与书名：`17px` 到 `19px`
- 正文编辑区：`18px` 左右，行高 `1.82` 到 `1.9`
- 元信息：`12px` 到 `13px`
- 引文与摘要：比元信息更清晰，但弱于正文

## 7.3 中文长文规则

- 正文最大行宽控制在适合中文阅读的范围，不让单行过长
- 不使用首行缩进，依靠段间距建立节奏
- 段落间距、引用块和小标题之间形成清楚但克制的节拍
- 标题、正文、引文、注释之间必须有明确角色差异

## 8. 色彩与材质

- 继续沿用以米白、纸白、灰褐、鼠尾草绿为核心的低饱和系统
- 保留 clay 与 gold 作为弱强调，只出现在引用题眼、选中提示或少量重点信息中
- 背景可有非常轻微的层次变化，但不得出现强烈彩色渐变
- 阴影应短而轻，主要用于纸页和浮层，不制造漂浮炫技感
- 面板、卡片、纸页的关系应是“材质差异”，不是“色块对撞”

## 9. 组件级设计

## 9.1 Sidebar

- 从“按钮列表”收敛成“索引目录”
- 品牌区更安静，保留识别但降低图标和按钮的突兀感
- 导航项通过细色条、淡底色、字重变化表达选中态
- hover 只允许轻微底色、边框和 `1px` 以内位移变化
- 底部统计与主题切换保留，但整体做得更像工具抽屉尾部信息

## 9.2 Workbench Header

- 标题保留衬线主标题，但弱化动作按钮的干扰
- 将“Reading Knowledge Desk”这类辅助文案处理成更安静的题眼
- 操作按钮维持清晰，但主次差更明确，避免与标题抢视线

## 9.3 书籍卡片

- 从“功能卡片”调整为“索引卡”
- 封面存在但退后，不允许压过书名和摘要
- 书名、作者、最近更新时间、记录数量、摘要建立更平衡的信息层次
- 选中态使用左侧细线、轻底色或边框加强，不使用厚阴影

## 9.4 笔记卡片

- 标题、引文、摘要之间拉开呼吸感
- 引文更像可回读的纸条片段，正文摘要更像思考线索
- 操作按钮默认退后，仅在 hover 或卡片尾部安静出现

## 9.5 右侧详情面板

- 从“信息面板”收敛为“边注页”
- 区块边界保留，但减少后台表单感
- 当前书籍、选中记录、主题标签三类信息分段清晰
- 收藏与打开编辑器等操作降低存在感，但保持明确可达

## 9.6 编辑页顶栏

- 顶栏只保留必要控制：返回、当前归属、保存状态、收藏、导出
- 保存状态文案应低干扰，不使用强弹出提示
- 面包屑和记录类型更像页眉提示，而不是管理工具栏

## 9.7 编辑纸页

- 纸页是编辑页的最高权重区域
- 纸页边缘、阴影、留白和最大宽度共同建立真实写作载体感
- 标题输入框要表现为文稿标题，而不是普通表单输入
- 摘录区与正文区必须明显区分：
  - 摘录区像引文纸条，有轻底色或轻框定
  - 正文区最干净，保证长文写作连续性

## 9.8 EditorToolbar

- 继续保留当前能力，不改交互模型
- 视觉上收敛成“短、轻、近”的器具带
- 激活态采用细腻底色与边框变化，不用强色块
- 工具条与纸页之间保持近距离，但不能压迫标题区

## 9.9 RichEditor 内容样式

- 正文段落、标题、引用、代码块、表格、图片风格全部统一到纸页系统
- 引用块重点突出阅读感，而非组件感
- 代码块和表格保留清晰边界，但降低技术后台视觉噪声
- placeholder 文案更短、更安静，像轻提示而不是说明书

## 9.10 编辑页右侧边注整理区

- 保留现有字段：书籍、类型、页码、标签、主题
- 将字段显示处理成“注释条目”，不是“后台配置表单”
- 标签与主题选择的视觉要更接近知识索引签
- 提示文案保留，但弱化说明口吻，强化思考引导口吻

## 10. 交互与状态

- hover：只提示可操作，不表演
- selected：通过细色条、轻底色、字重表达当前上下文
- focus：输入框与编辑器焦点要清晰，但不能打断写作
- saving：沿用现有自动保存节奏，文本提示使用“保存中 / 已保存 / 保存失败”一类低干扰状态
- empty：空状态文案保持简短安静，并提供唯一主操作
- loading：避免全屏阻断式加载，尽量维持原有布局骨架
- disabled：仍需可读，必要时通过提示文案说明原因

动画统一控制在 `120ms` 到 `180ms`，并遵守 `prefers-reduced-motion`。

## 11. 结构与实现边界

本次实现优先基于现有结构完成：

- `src/App.jsx`
- `src/components/Sidebar.jsx`
- `src/components/Workbench.jsx`
- `src/components/MetadataPanel.jsx`
- `src/components/EditorPage.jsx`
- `src/components/EditorToolbar.jsx`
- `src/components/RichEditor.jsx`
- `src/styles.css`

允许的结构优化：

- 当 `Workbench.jsx` 中书籍卡片和笔记卡片的视觉逻辑继续膨胀时，提取为更聚焦的小组件
- 当编辑页右侧字段区和纸页区样式耦合过重时，可做轻量拆分

不允许的结构变更：

- 不新增大型 UI 组件库
- 不重写应用状态管理方式
- 不改写现有后端接口与编辑器序列化逻辑

## 12. 数据流与行为约束

- 保留现有 `App -> Workbench / MetadataPanel / EditorPage` 的数据流
- 保留现有编辑页自动保存行为与 500ms 延迟节奏
- 不改变 `note`、`book`、`theme`、`tag` 数据结构
- 不改变现有收藏、导出、切换主题、打开编辑器的业务语义
- 样式优化不应引入新的内容丢失风险或保存时序问题

## 13. 错误处理与稳定性

- 现有加载失败和保存失败状态保留
- 视觉优化后，错误状态仍需与正常状态有足够区分，但不使用刺眼大面积警示色
- 当编辑器保存失败时，只强化状态可见性，不改动当前自动保存机制

## 14. 验证方案

实现完成后至少执行以下检查：

- `npm run build`
- `npm test`
- 桌面主场景检查：工作台三栏、编辑页纸页、右侧边注区
- 窄屏检查：确认布局折叠后仍可读、可编辑、可操作
- 主题检查：浅色与深色都保持书房感和可读性

## 15. 验收标准

- 打开工作台时，用户第一感受是安静、克制、有纸面与书写气质的知识工作台
- 三栏结构稳定，但中栏和编辑器明显成为主舞台
- 编辑页的标题、正文、引文、边注层次清楚，长时间阅读与写作更舒适
- 左右栏存在感降低但秩序更强，不再抢占主要视觉注意力
- 标签、卡片、按钮、保存状态、空状态和 hover 细节统一为同一套语言
- 不引入新的技术依赖，不破坏现有保存、导出和浏览流程

## 16. 实施建议

推荐实现顺序：

1. 先统一 token、背景、边框、阴影和排版基线
2. 再调整工作台三栏与卡片层次
3. 再精修编辑页纸页、工具条、引文区和边注区
4. 最后修状态细节、响应式和深色主题
